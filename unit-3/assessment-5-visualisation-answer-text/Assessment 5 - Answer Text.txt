Q1. Give a general description of the purpose of Matplotlib and Seaborn, and describe their features (5.1, 5.2).

Matplotlib

The purpose...

Matplotlib is a Python library used to create visualisations such as line charts, bar charts, scatter plots and histograms. Its purpose is to turn numerical data into charts so that patterns, comparisons and trends can be understood more clearly than by reading raw values alone. In AI programming, this is important because data normally needs to be checked and explored before it is used in modelling, reporting or assessment evidence.

In my Unit 3 work, Matplotlib would directly support the NumPy and Pandas evidence I have already produced. In Assessment 3, I created a NumPy ndarray called og_scores, saved it as .txt, .csv and .npy files, and uploaded the notebook and saved files to the OG-Website GitHub repository. Matplotlib could turn the og_scores values into a bar chart or line chart. In Assessment 4, I created the Pandas DataFrame og_dataframe with columns such as Task, Library, Status and Score. Matplotlib could visualise the Score column by Task so the assessor can see the evidence data as a chart as well as code output.

This fits the OG Web.site branded evidence approach because the finished visualisation could be shown in Jupyter Notebook, captured as a screenshot, linked through GitHub and placed into the branded evidence collection document.

Features...

Matplotlib has several useful features for Python evidence work. It supports many chart types, including line plots for trends, bar charts for category comparisons, scatter plots for relationships, histograms for distributions and pie charts for proportions. This means it can be used with different types of data depending on what needs to be explained. For my Unit 3 work, a bar chart would suit the og_dataframe Score column because it would compare the scores for Notebook, Screenshots and GitHub Upload.

Another key feature is control over the appearance of the chart. Matplotlib allows the programmer to add a title, x-axis label, y-axis label, legend, colours, markers, grid lines and figure size. This matters for my OG Web.site evidence because the chart should look clear and professional inside the branded evidence pack, not like an unexplained test output. Matplotlib also works with Python lists, NumPy arrays and Pandas DataFrames, so it links directly to the Assessment 3 NumPy ndarray and Assessment 4 Pandas DataFrame work. It can display charts inside Jupyter Notebook and export them as image files, which supports screenshots, GitHub evidence and Word document evidence collection.

Seaborn

The purpose...

Seaborn is a Python visualisation library built on top of Matplotlib. Its purpose is to make statistical charts easier to create, easier to read and better styled by default. It is especially useful with Pandas DataFrames because Seaborn can use named columns directly when creating a chart. This makes it practical when the data already has meaningful fields such as Task, Library, Status and Score.

In my Unit 3 Assessment 4 evidence, the og_dataframe table contains exactly that kind of structured data. Seaborn could use Task on one axis and Score on the other to show how the assessment evidence items compare. It could also group or colour data using a category such as Status or Library. This would make the Pandas evidence easier to understand because the chart would show the pattern visually instead of relying only on printed table output.

For the OG Web.site workflow, Seaborn would help create cleaner visual evidence from the data I have already produced. A Seaborn chart could be created in Jupyter Notebook, saved or screenshotted, uploaded to the GitHub evidence folder and placed into the OG Web.site branded evidence document.

Features...

A main feature of Seaborn is that it has attractive default styles and colour palettes. This means the charts often look cleaner and more professional without needing as much manual formatting. That is useful for my OG Web.site submission because evidence screenshots and charts should look organised, readable and suitable for a branded document. Seaborn also includes common statistical chart types such as bar plots, line plots, scatter plots, box plots, violin plots, distribution plots, pair plots, regression plots and heatmaps.

Another feature is its close integration with Pandas. Seaborn can take a DataFrame and use column names directly for the x-axis, y-axis, hue, category or grouping. This would work well with the Assessment 4 og_dataframe because the chart code could refer to actual column names such as Task, Status and Score. That makes the code easier to explain in a notebook and easier to evidence in screenshots. Seaborn can also show patterns between variables and differences between categories, which is useful when checking data before using it in AI programming. Because Seaborn is built on Matplotlib, Matplotlib can still be used afterwards to adjust the chart title, labels, layout and saving options.

Q2. Describe the features of good visualisation (5.3).

Good visualisation should be clear, accurate, relevant and easy to interpret. Clarity is important because the viewer should be able to understand the main message without guessing what the chart means. In my OG Web.site evidence pack, this would mean using a clear title, readable labels, sensible axis names and a layout that supports the assessment evidence. If I visualised the Assessment 4 og_dataframe, the chart should make it obvious what Task, Status and Score mean, rather than leaving the assessor to work it out from unclear labels.

Accuracy is also important. A visualisation should represent the data honestly and should not distort the values. Axis scales should be sensible, labels should match the data, and values should not be hidden or exaggerated. If I used Matplotlib or Seaborn to visualise the Assessment 3 og_scores ndarray, the chart would need to match the actual values saved in the .txt, .csv and .npy files. If I visualised the Pandas og_dataframe, the chart would need to match the values printed in the Jupyter Notebook and uploaded to GitHub.

Relevance is another feature of good visualisation. The chart should answer the question being asked and support the evidence, not just be added for decoration. A bar chart would be relevant for comparing the Score values in og_dataframe. A line chart would be relevant if I was showing progress over time. A scatter plot would be relevant if I was comparing two numerical values. The chart type should match the data and the purpose of the evidence.

Accessibility and consistency also matter. Colours should have enough contrast, text should be readable, and important information should not rely on colour alone. The chart should use a consistent style so it fits with the OG Web.site branded evidence collection, screenshots and GitHub links. Overall, good visualisation should make the data truthful, readable and useful. In my work, it should help the assessor understand the NumPy and Pandas evidence faster than reading the raw notebook output alone. A good visualisation should also be properly evidenced. If I created a chart from og_scores or og_dataframe, I would need the notebook cell, the chart output, the saved or uploaded file, and the GitHub link to support it. That would keep the visualisation connected to the same OG Web.site evidence standard I used in the NumPy ndarray and Pandas assessment work. It would also make the submission easier to check.
