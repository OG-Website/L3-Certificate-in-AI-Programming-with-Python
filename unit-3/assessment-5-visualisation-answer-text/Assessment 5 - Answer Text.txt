Q1. Give a general description of the purpose of Matplotlib and Seaborn, and describe their features (5.1, 5.2).

Matplotlib

The purpose...

Matplotlib is a Python library used to create data visualisations such as line graphs, bar charts, scatter plots, histograms and other figures. Its purpose is to turn numerical or structured data into a visual format so patterns, comparisons, trends and unusual values can be understood more easily than by reading raw values alone. In AI programming, this matters because data is normally checked and explored before it is used in a model or submitted as evidence.

In my OG Web.site workflow, Matplotlib would support the same evidence approach I have used across Unit 2 and Unit 3. My work is not just loose code; it is organised through branded evidence documents, GitHub folders, screenshots, notebooks and clear submission notes. Matplotlib would help by turning the numerical data created in NumPy or organised in Pandas into charts that can be placed inside the OG Web.site branded evidence pack. For example, the Pandas Assessment 4 DataFrame could be visualised as a bar chart showing task scores or evidence status by category. This would make the evidence easier for the assessor to read because the data would be shown visually as well as in code output.

Features...

One important feature of Matplotlib is that it gives detailed control over how a visualisation looks. A programmer can set the chart title, x-axis label, y-axis label, legend, colours, marker styles, line styles, figure size and axis limits. This is useful for my OG Web.site evidence because the chart can be made clear, labelled and suitable for a branded document instead of looking like a rough test output.

Matplotlib also supports a wide range of plot types. It can create line plots for trends, bar charts for category comparisons, scatter plots for relationships, histograms for distributions and many other chart types. It works with Python lists, NumPy arrays and Pandas DataFrames, which links directly to my Unit 3 work. In Assessment 3 I created and saved NumPy ndarray data, and in Assessment 4 I created Pandas Series and DataFrames. Matplotlib can use that kind of data to create visual evidence. It also works inside Jupyter Notebook, so the code and chart output can appear together. That would fit the way I have been building evidence: code, output, screenshots, GitHub links and OG Web.site branded documents all supporting each other.

Seaborn

The purpose...

Seaborn is a Python data visualisation library built on top of Matplotlib. Its purpose is to make statistical visualisations easier to create, easier to read and better styled by default. It is especially useful with Pandas DataFrames because Seaborn can use column names directly when creating charts. This makes it practical for structured data where each column has a meaning, such as task, library, score, status or category.

In my OG Web.site evidence workflow, Seaborn would be useful when the data needs to look professional quickly and when the visualisation is about patterns rather than just a simple chart. For example, if I had a Pandas DataFrame recording assessment tasks, scores, completion status or evidence types, Seaborn could help show differences between categories or relationships between values. This supports AI programming because visual exploration helps check whether the data makes sense before it is used further. It also supports the branded submission approach because the finished chart can be captured as evidence and included with the notebook, GitHub folder and OG Web.site document.

Features...

A main feature of Seaborn is that it provides attractive default styles and colour palettes. This means charts often look cleaner and more professional with less manual formatting. That is useful for my OG Web.site work because the visual evidence should look like part of a proper submission pack, not just a plain screen dump. Seaborn also includes high-level plotting functions for common statistical charts, including distribution plots, categorical plots, regression plots, box plots, violin plots, pair plots and heatmaps.

Another useful feature is its close integration with Pandas. Seaborn can take a DataFrame and use named columns for the x-axis, y-axis, colour grouping or category grouping. This keeps the code readable because the chart refers to meaningful column names. It would work well with the type of DataFrame evidence I created in Assessment 4, where columns such as Task, Library, Status and Score describe the data clearly. Seaborn can also show statistical information such as confidence intervals in some plots. Although Seaborn is built on Matplotlib, Matplotlib can still be used afterwards to adjust titles, labels, figure size and saving options. Together, they would let me create charts that are both quick to produce and suitable for OG Web.site branded evidence.

Q2. Describe the features of good visualisation (5.3).

Good visualisation should be clear, accurate, relevant and easy to interpret. Clarity is important because the viewer should be able to understand the main message without guessing what the chart means. In my OG Web.site evidence packs, this would mean using a clear title, readable labels, sensible axis names and a layout that supports the evidence instead of distracting from it. If a visualisation is showing assessment task scores, evidence progress or category comparisons, the assessor should be able to understand the point quickly.

Accuracy is another key feature. A visualisation should represent the data honestly and should not distort the values. Axis scales need to be sensible, labels need to match the data, and missing or unusual values should not be hidden in a misleading way. This matters in AI and data work because decisions may be made from the visual evidence. If I used Matplotlib or Seaborn to visualise the NumPy or Pandas work from Unit 3, the chart would need to match the actual data shown in the notebook and GitHub files.

Relevance is also important. A good visualisation should answer the question being asked and avoid unnecessary decoration. The chart type should match the data. A bar chart would suit comparisons between assessment tasks or evidence categories. A line chart would suit change over time. A scatter plot would suit relationships between two numerical values. For OG Web.site branded evidence, the visualisation should support the explanation in the answer sheet and evidence collection, not just be included because it looks good.

Accessibility and consistency are also part of good visualisation. Colours should have enough contrast, and important information should not rely on colour alone. Text should be large enough to read, and charts inside a document should follow a consistent style. This matters for my submissions because the evidence is organised through branded documents, screenshots and GitHub links. The visualisation should fit that professional structure. Overall, a good visualisation presents data in a way that is truthful, readable, relevant and useful for explaining the point being made.
